# pj — periodic jobs on launchd

A tiny framework for scheduling scripts (especially `claude -p` automations) on
macOS via `launchd`. You describe a job once; `pj` templates the launchd plist
and a wrapper script, keeps everything version-controlled, and runs each job
with your **full interactive shell environment** loaded.

```
pj add triage --at 09:00 --cd ~/code/repo --run 'claude -p "daily triage" --permission-mode dontAsk'
pj list
pj logs triage
```

Run `pj` with no command for the full help, or `pj add` with no arguments for an
interactive wizard.

## Why it exists

`launchd` jobs start with a near-empty environment and never load your
`~/.zshrc`. `claude` lives at `~/.local/bin/claude`, and your toolchain
(rbenv/pyenv/volta/cargo/gcloud) is all on a PATH that only your interactive
shell builds. `pj` bridges that gap and removes the XML/`launchctl` busywork.

macOS only — it drives `launchctl` in your per-user GUI domain.

## Install

`pj` is a [uv](https://docs.astral.sh/uv/) inline script — no venv to manage.
Clone the repo and put the `pj` CLI on your `PATH`:

```
git clone git@github.com:mitio/pj.git ~/.local/share/pj
ln -sf ~/.local/share/pj/pj ~/bin/pj      # anywhere on your PATH
```

`pjkit.py` must sit next to the `pj` script (it does in a clone); the symlink is
followed to find it.

### Where jobs live — `PJ_JOBS_DIR`

Jobs are stored under `PJ_JOBS_DIR`, defaulting to `~/.pj/jobs`. Point it at a
tracked directory to version-control your jobs:

```
export PJ_JOBS_DIR="$HOME/dotfiles/jobs"
```

### Vendoring into a dotfiles repo (git submodule)

`pj` is designed to live as a submodule of a dotfiles repo, with the jobs
tracked alongside the rest of your config. From your dotfiles repo:

```sh
# 1. add pj as a submodule
git submodule add git@github.com:mitio/pj.git pj

# 2. symlink the CLI onto your PATH (in your install/setup script)
ln -sf "$HOME/dotfiles/pj/pj" "$HOME/bin/pj"

# 3. tell pj to keep jobs in the repo (in ~/.zshrc or equivalent)
export PJ_JOBS_DIR="$HOME/dotfiles/jobs"
```

On a fresh machine, `git submodule update --init` pulls `pj`, the symlink puts it
on `PATH`, and `PJ_JOBS_DIR` points it back at your tracked `jobs/`. To update
the tool later: `git -C pj pull && git add pj && git commit -m 'bump pj'`.

## How a job runs

`launchd` executes:

```
/bin/zsh -i -c "source <job>/wrapper.sh"
```

- `-i` makes zsh source your `~/.zshrc`, so the **full environment** is present.
- `source` (not `exec`) runs the wrapper *inside* that interactive shell, so your
  **aliases and shell functions** work too — not just exported `PATH`/vars.
- `wrapper.sh` `cd`s into the job's directory (so that folder's `.claude/skills`
  and settings apply) and runs your command, logging a start/exit banner.

## Surviving reboots

`launchd` only auto-loads jobs at login from a standard `LaunchAgents` directory.
A plain `launchctl bootstrap` is in-memory and gets dropped on logout/reboot — so
enabling a job also **symlinks its plist into `~/Library/LaunchAgents`** (pointing
back at the tracked copy in `$PJ_JOBS_DIR`). That symlink is what brings jobs back
after a restart. `pj disable` / `pj rm` remove it; `pj doctor` flags any enabled
job that isn't linked (`pj apply <name>` re-links it).

## Layout

```
$PJ_JOBS_DIR/
  <name>/
    job.toml                       # the job definition — source of truth (tracked)
    wrapper.sh                     # generated; what actually runs (tracked, editable)
    com.dimitrov.pj.<name>.plist   # generated launchd config (tracked)
    logs/{out,err}.log             # runtime logs (gitignore these)
```

## Source-of-truth model

`job.toml` is authoritative. `wrapper.sh` and the plist are **generated from it**
and carry a `generated by pj` marker:

- `pj apply <name>` regenerates both from `job.toml` and reloads.
- Delete the marker line in `wrapper.sh` (or the plist) to **adopt** the file —
  `pj` then leaves it alone. Use `pj apply --regen-wrapper` / `--force` to rebuild.

So: edit `job.toml` for routine changes; adopt the wrapper for custom logic.

## Commands

| Command | Does |
|---|---|
| `pj add [name] [flags]` | Add a job. No name (or `-i`) → interactive wizard. |
| `pj list` | Table of jobs: schedule, enabled, launchd state, last exit. |
| `pj show <name>` | Config + recent logs. |
| `pj edit <name>` | Edit `job.toml` in `$EDITOR`, then apply. |
| `pj edit-wrapper <name>` | Edit `wrapper.sh` in `$EDITOR`. |
| `pj apply <name>` | Regenerate wrapper+plist from `job.toml` and reload. |
| `pj enable` / `pj disable <name>` | Load / unload (files kept). |
| `pj run <name>` | Run now (`launchctl kickstart`). |
| `pj logs <name> [-f] [-n N]` | Tail stdout+stderr. |
| `pj status <name>` | launchd state, last exit, run count. |
| `pj rm <name> [-f]` | Unload and delete the job folder. |
| `pj doctor` | Check the environment is ready for unattended runs. |

## Scheduling (friendly flags)

```
pj add x --at 09:00                 # daily at 09:00
pj add x --at 09:00 --at 18:30      # twice daily
pj add x --at 09:00 --days mon,fri  # weekdays only
pj add x --at 'mon,fri 18:00'       # weekdays embedded in the time
pj add x --every 30m                # every 30 minutes (also: 2h, 90s)
pj add x ... --run-at-load          # also run once when (re)loaded / at login
```

Everything translates to launchd `StartCalendarInterval` / `StartInterval`.

## Running `claude` unattended

There is no human to approve permission prompts, so set a non-interactive mode in
your command. Safest is `--permission-mode dontAsk` (auto-denies anything not in
that folder's `.claude/settings.json` `permissions.allow`); `--permission-mode
acceptEdits` is more permissive; `--dangerously-skip-permissions` bypasses all
checks (trusted dirs only).

Auth: your subscription credentials in `~/.claude/.credentials.json` are a file
(not Keychain), so jobs read and refresh them fine. For a long-lived token
instead, run `claude setup-token` and export `CLAUDE_CODE_OAUTH_TOKEN` in your
shell.

## Caveats

- **Interval jobs missed while asleep** coalesce into a single run on wake.
- **Calendar weekdays** apply to all listed times (one weekday set per job). For
  genuinely different days-per-time, make two jobs.
- Running `wrapper.sh` directly (`./wrapper.sh`) only gets exported env, not
  zshrc aliases/functions — that fidelity comes from the launchd `zsh -i` path.
